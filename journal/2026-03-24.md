# 2026-03-24 (Day 23)

## 오늘 한 것

오늘은 로그와 지표를 함께 보고 원인을 판단하는 실무 장애 분석 연습을 했다.

기존에는 CPU, Disk, Memory, VMware 지표를 중심으로 분석했다면  
오늘은 실제 로그를 통해 성능 문제가 아니라 장애 상황을 구분하는 연습을 했다.

특히 I/O error 로그를 통해 단순 디스크 지연이 아니라  
실제 디스크 장애 상황이라는 것을 이해했다.

또한 메모리 부족 상황에서 OOM 로그가 발생하면  
커널이 프로세스를 강제로 종료하는 상황이라는 것을 확인했다.

마지막으로 OS 지표는 정상인데도 서버가 느린 경우  
애플리케이션 레벨(DB connection pool) 문제일 수 있다는 것을 배웠다.

---

## 기억할 것

- I/O error는 성능 문제가 아니라 디스크 장애
- OOM 로그는 메모리 부족으로 인한 프로세스 강제 종료
- SWCUR 높으면 매우 위험한 상태 (Swap 과다)
- OS 지표가 정상인데도 느리면 애플리케이션 문제 의심
- DB connection timeout + pool exhausted → DB 연결 부족 문제

---

## 느낀 점

오늘은 지표만 보는 것이 아니라 로그를 같이 봐야  
정확한 원인을 판단할 수 있다는 것을 느꼈다.

특히 성능 문제와 장애는 완전히 다른 개념이라는 점이 인상적이었다.

또 OS가 정상이라고 해서 문제가 없는 것이 아니라  
애플리케이션이나 DB에서 문제가 발생할 수 있다는 것을 이해하게 되었다.

이제는 단순히 “느리다”가 아니라  
어디 레벨에서 문제가 발생했는지 구분하는 것이 중요하다는 걸 느꼈다.
